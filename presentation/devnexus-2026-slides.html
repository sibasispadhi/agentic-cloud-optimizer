Pattern 1: Observe → Reason → Enforce

🔍 OBSERVE (Deterministic)
p95/p99 metrics · SLO breach detection · JMX/GC telemetry
▼
🧠 REASON (AI-Powered)
LLM analyzes multi-dimensional metrics · Explainable recommendations
▼
⚡ ENFORCE (Bounded Actions)
±25% max change · Cooldown windows · Auto-rollback

Why This Works

  • Observe is deterministic — no AI hallucinations
  • Reason uses LLM as advisor, not decision-maker
  • Enforce constrains actions — safe by design
  • Each layer is independently testable
  • Works anywhere: local, Docker, K8s, VMs
LLM proposes → Policy constrains → System verifies → Rollback exists

SLO Breach Detection: The Trigger

When should the agent activate? SLO breach detection answers that.

SloPolicy policy = SloPolicy.builder()
    .targetP99Ms(100.0)           // p99 must be under 100ms
    .breachThreshold(1.2)         // Breach if > 120ms (20% over)
    .breachWindowIntervals(3)     // Must breach 3 consecutive checks
    .build();

// In the optimization orchestrator:
sloBreachDetector.recordMetric(baseline);
if (sloBreachDetector.isBreached(policy)) {
    log.warn("🚨 SLO BREACH: " + detector.getBreachReason());
    // → Trigger autonomous agent analysis
}
✅ Normal Operation
p99 = 85ms → Under 120ms threshold
Status: SLA COMPLIANT
❌ Breach Detected
p99 = 135ms for 3 intervals → Over 120ms
🚨 Triggering agent analysis!

Pattern 2: Bounded Action Space

The agent is powerful — but constrained. Safety first.

❌ Without Bounds

LLM: "Reduce heap from 8GB to 1GB"
     → -87.5% change
     → System crashes
     → Outage at 2 AM
     → Incident report time 📝

✅ With Bounds

LLM: "Reduce heap from 8GB to 1GB"
ARG: ❌ REJECTED (exceeds ±25% limit)

LLM: "Reduce heap from 8GB to 6GB"
ARG: ✅ APPROVED (-25%, within bounds)
     → Change applied safely
     → p99 improved 58%!
Safety constraints: Max ±25% parameter delta · 10-min cooldown between changes · Auto-rollback if p99 worsens >10% · Forbidden parameter list

Pattern 3: LLM as Advisor

The LLM advises. Deterministic code decides.

What the LLM Does

  • Interprets multi-dimensional metrics
  • Correlates heap usage, GC pauses, p99
  • Proposes bounded recommendations
  • Explains reasoning (audit trail)
// Spring AI + Ollama integration
@Service
public class SpringAiLlmAgent {
  AgentDecision decideWithHeapAnalysis(
    RunResult baseline,
    int currentConcurrency
  );
}

What the LLM Does NOT Do

  • ❌ Make final decisions
  • ❌ Apply changes directly
  • ❌ Bypass safety constraints
  • ❌ Access production systems
🔑 Key principle: LLMs hallucinate. Governance doesn't.
Deterministic policy enforcement is the safety net.

Reusable in any Java microservice — not ACO-specific.

Key Takeaways

Tail latency is where SLA breaches and customer pain live.
2. LLM as advisor, not dictator.

3. Bounded autonomy is the sweet spot.
±25% limits, cooldowns, and auto-rollback make self-optimization safe.
4. Start small, evolve incrementally.
